# 05 Prompt 工程

本讲介绍如何通过 **Prompt 设计** 控制大模型的输出风格、格式与推理方式。同一用户问题，搭配不同的 system 设定、示例或格式要求，会得到截然不同的回答。本 Demo 对同一问题使用多种 prompt 风格（默认、角色设定、格式约束、Few-shot、CoT），并在运行时展示每种风格对应的**完整 prompt** 与模型回复，便于对比理解。

---

## 核心概念

### 1. System prompt 设计

**是什么**：System prompt 是发给模型的一条「系统级」消息（通常 role 为 system），用来设定任务目标、行为规则、回答范围或禁忌。模型会优先遵循这些设定，再结合 user/assistant 内容生成回复。

**为什么要有这个概念**：不设 system 时，模型按默认通用行为回答，风格和边界不可控。通过 system 明确「你要做什么、不要做什么」，可以在不微调的前提下，让同一模型在不同产品里扮演不同角色、输出不同风格，是成本最低的「行为定制」手段。

### 2. 角色设定

**是什么**：在 system（或首条 prompt）里明确描述「你是谁」，例如「你是一位技术老师」「你是客服助手」。角色设定会影响语气、用词、详略和立场，使回复更一致、更符合场景。

**为什么要有这个概念**：模型没有固定「人设」，同一问题可能答得过于随意或过于正式。用角色限定身份与场景，能统一口吻、减少无关发散，提升可预测性和用户体验。

### 3. Few-shot（少样本示例）

**是什么**：在对话中插入一两组「用户问题 + 模型回答」的示例（即 user/assistant 对），让模型模仿示例的格式、长度或风格来回答当前问题。不额外训练，仅通过上下文中的示例做 in-context learning。

**为什么要有这个概念**：有些格式或风格用文字很难描述清楚（如「用要点+一句话解释」）；直接给一个示例，模型更容易对齐。Few-shot 适合固定输出格式、特定话术、或需要严格仿写风格的场景。

### 4. CoT（Chain-of-Thought，链式思考）

**是什么**：在 prompt 中要求模型「先一步步推理再给答案」，例如先写「思考：…」再写「答案：…」。模型会先生成中间推理步骤，再给出最终结论，从而提升多步推理、数学、规划类问题的正确率。

**为什么要有这个概念**：很多任务需要多步推理，若直接要最终答案，模型容易跳步或出错。显式要求「先思考再作答」，能激活模型的推理过程，减少逻辑错误，也便于人类检查推理链。

### 5. 输出格式约束

**是什么**：在 prompt 中明确规定回答的结构，例如「请按以下格式：1. … 2. … 3. …」「请输出 JSON：{ "a": ..., "b": ... }」「请用 Markdown 列表」。模型会尽量按该结构生成，便于后续解析或展示。

**为什么要有这个概念**：自由文本难以被程序稳定解析；若下游要做筛选、展示、入库，需要固定格式。通过格式约束，可以在不写复杂后处理的前提下，得到结构化或半结构化输出。

## 本 Demo 做了什么

- 对同一用户问题（如「用三句话介绍 Python」）依次使用五种风格：默认（无 system）、System+角色设定、输出格式约束、Few-shot 示例、CoT。
- 每种风格会先打印 **【本风格使用的 Prompt】**（完整 system/user 及示例内容），再调用 **core.llm** 得到 **【模型回复】**，便于对照学习。

运行：在项目根目录执行 `python -m demos.05_prompt.run`。学习站内点击「运行 Demo」会执行该脚本并展示上述输出。
