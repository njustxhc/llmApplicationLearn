# 17 错误处理与重试

本讲演示如何对 LLM 请求做**重试**与**退避**，在瞬时故障（网络抖动、限流、服务短暂不可用）时提高调用成功率。

---

## 核心概念

### 1. 重试（Retry）

**是什么**：一次请求失败后，不立即放弃，而是隔一段时间再次发起相同请求，最多重试 N 次。适用于临时性错误（如超时、5xx）。

**为什么要有这个概念**：网络与远程 API 存在偶发失败；重试能以较小成本显著提高成功率，是生产环境的常见做法。

### 2. 指数退避（Exponential Backoff）

**是什么**：重试间隔随次数增加而延长，例如第 1 次等 1 秒、第 2 次等 2 秒、第 3 次等 4 秒。避免在服务恢复前「扎堆」重试，加重压力。

**为什么要有这个概念**：若固定间隔且较短，故障期间会持续打满请求，可能加剧限流或延迟恢复；退避能兼顾「尽快重试」与「给服务喘息时间」。

### 3. 可重试与不可重试错误

**是什么**：**可重试**：超时、连接错误、5xx、部分 429（限流）。**不可重试**：4xx 业务错误（如参数错误、鉴权失败）、内容过滤拒绝等，重试通常无意义。

**为什么要有这个概念**：只在「可重试」错误上重试，避免无谓请求与逻辑混乱；对不可重试错误应直接返回或降级。

## 本 Demo 做了什么

### 基础用例

- 封装一个带重试的 chat 调用：在请求失败（如超时、5xx）时按指数退避进行有限次重试，成功则返回结果，最终仍失败则抛出或返回错误信息。演示如何让单次 LLM 调用在瞬时故障下更健壮。
- 运行：点击「运行基础 Demo」或执行 `python -m demos.17_error_retry.run`。

### 进阶用例

- 区分「可重试错误」与「不可重试错误」：例如 429 限流单独处理，4xx 客户端错误不重试；对可重试与不可重试分别打日志或返回不同错误类型，便于上层做熔断或降级。
- 运行：点击「运行进阶 Demo」或执行 `python -m demos.17_error_retry.run_advanced`。
