# 09 评估与评测（Eval）

本讲介绍如何对大模型输出做**自动评测**：构造测试问题与期望（如关键词、标准答案），调用模型作答后，用规则或模型再对回答打分、判对错，从而评估效果、回归测试或对比不同 prompt/模型。本 Demo 使用「问题 + 期望关键词」的简单评测：调用 **core.llm** 得到回答，再检查回答中是否包含期望关键词，统计通过率。

---

## 核心概念

### 1. 评估 / 评测（Evaluation）

**是什么**：对大模型输出进行质量评估，包括：是否切题、是否包含关键信息、格式是否正确、是否有幻觉等。评估可以是人工打分，也可以是自动规则（如关键词、正则）或另一套模型（LLM-as-judge）打分。

**为什么要有这个概念**：改 prompt、换模型、加 RAG 后，需要知道「是否变好、是否变差」。没有评估就无法迭代；自动评测能批量、可重复地给出指标，是迭代优化与回归测试的基础。

### 2. 测试集（Test Set）

**是什么**：一组预先准备好的「输入 + 期望」对，例如（问题, 期望关键词）、（问题, 标准答案）、（问题, 期望格式）。评测时对每条输入调用模型，再根据期望对输出进行判分或匹配。

**为什么要有这个概念**：没有固定测试集，每次改完只能「感觉一下」效果。固定测试集能在同一标准下对比不同版本，保证评测可复现、可比较。

### 3. 关键词 / 规则判分

**是什么**：用简单规则判断输出是否合格，例如：期望回答中包含若干关键词即判对、或必须包含某短语、或符合某正则。本 Demo 采用「期望关键词」：若模型回答中包含指定关键词之一或全部，则该项通过。

**为什么要有这个概念**：规则简单、可解释、不依赖额外模型，适合对「必须提到某概念」类需求做快速校验；缺点是难以衡量语义等价、表述多样的情况，可作为基线或第一道关卡。

### 4. LLM-as-Judge（模型作为裁判）

**是什么**：用另一个大模型对「问题 + 模型输出」进行打分或评判，例如判断相关性、事实性、是否有害等。裁判模型根据设定好的评分维度与标准给出分数或等级。

**为什么要有这个概念**：很多维度（如「回答是否得体」「是否全面」）难以用规则刻画。用 LLM 做裁判能覆盖语义层面的评估，更贴近人类判断，适合作为规则评测的补充或主评估手段。

### 5. 通过率与回归

**是什么**：在测试集上统计「通过条数 / 总条数」得到通过率；每次改代码或 prompt 后重新跑一遍，若通过率下降则可能发生回归，需排查原因。

**为什么要有这个概念**：单一指标便于监控与对比；回归测试能避免「修了 A 坏了 B」。把评测集成进开发流程，能提高迭代信心与交付质量。

---

## 本 Demo 做了什么

- 定义若干（问题, 期望关键词）用例，例如问题「Python 是什么」、期望包含「编程」「语言」等。
- 对每个问题调用 **core.llm** 得到回答，检查回答中是否包含期望关键词（可配置：包含任一即通过，或需全部包含）。
- 统计通过数量与通过率并打印，演示最简单的自动评测闭环。

运行：在项目根目录执行 `python -m demos.09_eval.run`。
