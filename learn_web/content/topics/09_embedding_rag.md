# 09 Embedding 与向量检索（RAG 深化）

本讲在 RAG 的基础上，引入 **向量表示** 与 **向量检索**：用 Embedding 把文本变成向量，在向量空间里做相似度检索，从而得到语义上更相关的片段，而不只依赖关键词。本 Demo 使用 **core.rag_doc**（当前为 TF-IDF 实现）模拟「检索 → 注入 → 生成」的流程；生产环境可替换为真实的 Embedding API 与向量库。

---

## 核心概念

### 1. Embedding（向量表示）

**是什么**：Embedding 是把一段文本（词、句、段）映射成一个固定长度的数值向量。语义相近的文本，其向量在空间中的距离较近（如余弦相似度高）；语义差异大的文本，向量距离较远。通常由专门的 Embedding 模型或 API 生成。

**为什么要有这个概念**：关键词匹配（如 TF-IDF）只能抓住「词是否出现」，无法表达「意思是否相近」。用向量表示语义后，可以用「向量相似度」做检索，支持同义表达、泛化查询，更适合开放域问答与知识库检索。

### 2. 向量检索（相似度搜索）

**是什么**：把文档库中的每段文本预先做成向量并存入**向量库**；查询时把用户问题也变成向量，在向量库中找出与查询向量最相似的若干条（如 top-k），对应的原文即为检索结果。相似度常用余弦相似度或内积衡量。

**为什么要有这个概念**：有了 Embedding 后，检索问题就变成「在向量空间里找最近邻」。向量检索能利用语义相似度，比纯关键词更鲁棒，是当前 RAG 系统中主流的检索方式。

### 3. 向量库（Vector Store）

**是什么**：专门存储向量并支持「按向量相似度快速检索」的数据库或索引结构，如 Milvus、Chroma、FAISS、pgvector 等。文档先切块、再 Embedding、再写入向量库；查询时对 query 做 Embedding，再在向量库中做 k-NN 或近似最近邻搜索。

**为什么要有这个概念**：暴力遍历所有向量算相似度在大规模数据下不可行。向量库提供索引与优化算法，能在百万、千万级向量上快速返回 top-k，是 RAG 可扩展性的基础。

### 4. TF-IDF 与向量检索的关系

**是什么**：TF-IDF 是一种基于词频与逆文档频率的文本表示（也可看作一种简单向量）；本 Demo 用 TF-IDF 在文档上做检索，模拟「检索 → 注入 → 生成」的 RAG 流程。真正做语义检索时，会改用 Embedding 向量 + 向量库。

**为什么要有这个概念**：在没有 Embedding API 或向量库的环境下，TF-IDF 能快速演示 RAG 的完整链路；理解「检索结果注入 prompt」的逻辑后，只需把检索模块换成 Embedding+向量库，即可升级为语义 RAG。

### 5. RAG 的「检索 + 生成」闭环

**是什么**：RAG 的完整流程是：用户问题 → 检索（关键词或向量）得到相关片段 → 将片段与问题一起注入 prompt → 大模型基于注入内容生成答案。检索质量直接决定模型「能看到什么」，从而影响答案的准确性与依据性。

**为什么要有这个概念**：检索与生成是 RAG 的两大支柱：检索负责「找对材料」，生成负责「组织成答案」。用好 Embedding 与向量检索，能显著提升「找对材料」这一步，是 RAG 深化的核心。

## 本 Demo 做了什么

### 基础用例

- 使用 **core.rag_doc**（当前为 TF-IDF）在本地文档上做「模拟向量检索」：对用户 query 与文档片段做相似度计算，取 top_k 片段后交给 **core.llm** 生成答案。演示「检索 → 注入上下文 → 生成」的 RAG 流程。
- 运行：点击「运行基础 Demo」或执行 `python -m demos.09_embedding_rag.run`。

### 进阶用例

- 用更大 top_k 做多段检索并合并，或演示「多段检索 + 合并」的流程；若环境支持，可接入真实 Embedding API 与向量库做语义检索。
- 运行：点击「运行进阶 Demo」或执行 `python -m demos.09_embedding_rag.run_advanced`。
