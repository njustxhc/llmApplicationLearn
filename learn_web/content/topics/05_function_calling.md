# 05 Function Calling（函数/工具调用）

本讲介绍 **Function Calling**：让大模型在回复中「声明」要调用的函数名和参数，由应用层解析并真正执行，再把执行结果塞回对话，让模型基于结果继续生成。这样模型就能间接完成查天气、算数、查库等操作。本 Demo 通过 **Prompt 约定**（如 `TOOL: 函数名 参数`）引导模型输出「调用意图」，再用 **core.llm** 完成对话与工具结果回填。

---

## 核心概念

### 1. Function Calling（函数调用）

**是什么**：Function Calling 指大模型在生成回复时，不仅输出自然语言，还会输出「要调用哪个函数、传什么参数」的结构化信息。应用解析后真正执行该函数，把结果作为新消息（如 assistant 的 tool 结果或 user 的「工具返回」）塞回对话，模型再根据结果生成最终回答。

**为什么要有这个概念**：模型不会真的执行代码或访问 API，只能通过「说出要调用什么」把意图交给应用。由应用执行并回填结果，既保证安全与可控，又让模型能完成需要实时数据或计算的任务（天气、计算、查库等）。

### 2. 工具描述与参数 Schema

**是什么**：在调用模型前，应用会把「可用工具」以某种形式告诉模型，例如：工具名、自然语言描述、参数名与类型（或 JSON schema）。模型根据这些描述，在回复中选择工具并填写参数。

**为什么要有这个概念**：模型需要知道「能调用什么、参数长什么样」，才能输出正确的函数名和参数。工具描述和 schema 是「模型与真实函数」之间的契约，描述越清晰，模型越容易选对工具、填对参数。

### 3. Prompt 约定 vs 原生 tools 参数

**是什么**：**原生 tools 参数**：API 支持在请求里传 `tools` 数组，模型在响应里通过 `tool_calls` 等字段返回结构化调用信息。**Prompt 约定**：不依赖 API 的 tools 字段，而是在 system/user 里用文字约定「请用 TOOL: 函数名 参数 的格式输出」，应用再解析模型文本得到函数名和参数。

**为什么要有这个概念**：部分 API 尚未支持或限制 tools 格式；用 Prompt 约定可以快速实现「模型输出调用意图」的闭环，便于理解和教学。生产环境在 API 支持时，可改为原生 tools，获得更稳定的解析与多工具支持。

### 4. 执行与结果回填

**是什么**：应用根据模型输出解析出「函数名 + 参数」后，在本地或远程执行该函数，得到返回值（如字符串、数字、JSON）。该返回值作为一条新消息（例如 user 消息：「工具返回：…」）追加到 messages，再请求模型生成下一步回复。

**为什么要有这个概念**：模型只有「看到」工具执行结果，才能基于结果总结、判断或继续规划。回填把「真实世界」的反馈纳入对话，形成「用户问 → 模型决定调工具 → 应用执行 → 结果回填 → 模型再答」的闭环。

## 本 Demo 做了什么

### 基础用例

- 在 system prompt 中约定：模型若需要计算或打招呼，请输出 `TOOL: add a b` 或 `TOOL: greet 名字` 等形式。
- 单轮流程：user 提问 → 调用 **core.llm** 得到回复 → 解析是否包含 TOOL 调用 → 若包含则执行对应函数（add/greet），将结果拼成消息再请求一次模型 → 输出最终回答。
- 演示「模型输出工具调用意图 → 应用解析并执行 → 结果回填」的单轮工具调用流程。
- 运行：点击「运行基础 Demo」或执行 `python -m demos.05_function_calling.run`。

### 进阶用例

- 在同一对话中支持多轮、多次 function call：维护 messages 历史，在每一轮中解析模型输出中的 TOOL 调用，依次执行并把结果回填，让模型综合多步工具结果再回答。
- 运行：点击「运行进阶 Demo」或执行 `python -m demos.05_function_calling.run_advanced`。
