# 19 上下文截断（Context Truncation）

本讲演示当**对话历史过长**时，如何通过**截断**（只保留最近 N 条消息）控制发给模型的上下文长度，避免超长报错或成本过高。

---

## 核心概念

### 1. 上下文窗口（Context Window）

**是什么**：模型单次请求能接受的最大 token 数（或消息条数）称为上下文窗口。超过后需截断、摘要或拒绝，否则 API 报错或行为异常。

**为什么要有这个概念**：多轮对话会不断追加历史，迟早会触顶；必须主动管理「哪些保留、哪些丢弃或压缩」，才能长期稳定运行。

### 2. 截断策略

**是什么**：常见策略包括：**保留最近 N 条**（本 demo）、**保留最近 N 个 token**、**保留首条 system + 最近若干轮**、或对更早的消息做**摘要**后再保留（见 07 记忆）。

**为什么要有这个概念**：不同策略在「保留近期细节」与「保留早期要点」之间取舍不同；按产品需求选择或组合策略。

### 3. 截断与摘要的配合

**是什么**：单纯截断会直接丢失被截掉的内容；若需保留「更早的概要」，可先对旧消息做 LLM 摘要，再用「摘要 + 最近 N 条原文」组成新上下文（06 多轮记忆 demo 中有类似逻辑）。

**为什么要有这个概念**：长会话既要控制长度又要尽量保留信息，摘要+截断是常用组合。

---

## 本 Demo 做了什么

### 基础用例

- 构造一段较长的模拟对话历史（如 10+ 轮），演示「保留最近 K 条消息」的截断策略：只将最近 K 条（如 5 条）拼成 messages 发给模型，避免超长上下文报错或成本过高。帮助理解滑动窗口式截断的实现方式。
- 运行：点击「运行基础 Demo」或执行 `python -m demos.19_context_truncate.run`。

### 进阶用例

- 在「最近 K 条」基础上扩展：例如「按 token 数截断」——拼 messages 时累计 token 估算，超过上限则不再追加更早的消息；或「摘要 + 截断」——对较早的 N 条先做摘要，再与最近 K 条原始消息一起组成 context。
- 运行：点击「运行进阶 Demo」或执行 `python -m demos.19_context_truncate.run_advanced`。
