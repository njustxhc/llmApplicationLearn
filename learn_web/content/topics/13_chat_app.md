# 13 综合对话应用（Chat App）

本讲是一个**综合 Demo**：把前面学过的「基础 API 与对话」「RAG」「MCP 工具」「联网搜索」整合在一个 Web 对话应用里。通过页面与大模型对话，并可勾选是否启用文档 RAG、MCP 天气、联网搜索，用于体会多能力组合的完整链路。

---

## 核心概念

### 1. 综合应用（Integrated Chat App）

**是什么**：在一个对话界面中同时接入多种增强能力：基础 LLM 对话、文档 RAG 检索、MCP 工具调用、联网搜索等。前端通过勾选项控制本次请求启用哪些能力；后端按勾选顺序注入对应上下文（如 RAG 片段、天气结果、搜索结果），再调用大模型生成回复。

**为什么要有这个概念**：单讲 RAG、MCP、搜索时是「单能力」demo；真实产品往往需要组合多种能力。综合应用演示了如何在同一套 messages 与 API 下，按需注入多种上下文，是前面各讲的集成与落地形态。

### 2. 上下文注入顺序与组合

**是什么**：当用户同时勾选「联网搜索 + MCP + 文档 RAG」时，后端会依次执行：先联网搜索并注入一条 system 消息 → 再若识别为天气问题则调 MCP 并注入一条 system → 再执行文档 RAG 检索并注入一条 system，最后加上用户消息一起发给 LLM。多条 system 共同构成「当前可用的外部知识」。

**为什么要有这个概念**：不同能力产生的上下文可能互补也可能重叠；注入顺序会影响模型优先看到哪类信息。理解顺序与组合，便于后续做优先级、去重或更复杂的编排（如 Agent 决策先查哪一类）。

### 3. 流式输出与前端展示

**是什么**：本应用支持流式（SSE）回复：后端使用 **core.llm** 的 `chat_stream` 逐 token 返回，前端通过 EventSource 或 fetch + ReadableStream 接收并实时追加到界面，形成「打字机」效果。

**为什么要有这个概念**：长回复时非流式等待时间长、体验差。流式能降低首字延迟、提升交互感，是对话类产品的常见需求；同时需处理 SSE 格式、断线重连与错误展示。

### 4. 可选工具与按需调用

**是什么**：用户通过勾选「联网搜索」「MCP 工具」「文档 RAG」决定本轮是否调用相应能力。后端根据勾选与用户消息内容（如是否包含「天气」）决定是否调 MCP、是否做 RAG、是否做搜索，再组装 messages。

**为什么要有这个概念**：不是每条消息都需要所有能力；按需调用能节省成本、减少延迟，也便于用户理解「这次回答用了哪些外部信息」。为后续 Agent「自主选工具」打基础。

## 本 Demo 做了什么

### 基础用例

- **Web 界面**：提供对话输入框与「联网搜索」「MCP 工具」「文档 RAG」三个勾选项；支持流式回复与多轮对话。
- **后端**：使用 **core.llm** 做对话与流式输出；使用 **core.rag_doc** 做文档检索并注入；使用 **demos.04_mcp.server** 的 get_weather 做天气查询并注入；使用 **ddgs** 做联网搜索并注入。演示如何将 RAG、MCP、联网搜索整合在一个前端里。
- **运行方式**：本 Demo 为**独立 Web 应用**，需单独启动。学习站内点击「运行 Demo」会提示启动命令与访问地址（http://127.0.0.1:5000）。

### 进阶用例

- 可选方向：在现有应用上增加多会话或会话持久化；或仅在学习站与文档中补充「进阶使用」说明（如如何配置 MCP、如何扩展工具）。当前前端仅提供单入口，进阶以文档扩展为主。

## 如何运行

在**项目根目录**执行：

```bash
python -m demos.13_chat_app.app
```

然后打开浏览器访问 **http://127.0.0.1:5000**。确保已配置 `ARK_API_KEY`；若使用 MCP 需先安装 `mcp`，使用联网搜索需安装 `ddgs`。
