# 17 Token 与成本意识

本讲建立**上下文长度**与**调用成本**的概念：用简单启发式估算「消息约等于多少 token」，理解控制上下文长度的意义。

---

## 核心概念

### 1. Token 与上下文长度

**是什么**：模型以 **token** 为单位处理文本，上下文有最大 token 数限制（上下文窗口）。中文约 1～2 字符/token，英文约 4 字符/token，因模型与分词方式而异。

**为什么要有这个概念**：超长上下文会触发截断或报错；且 token 数直接关联计费，控制长度即控制成本与可用范围。

### 2. Token 计数

**是什么**：在发送前或收到响应后，统计「输入 token 数」与「输出 token 数」。精确计数需使用模型对应的 **tokenizer**（如 tiktoken）或依赖 API 返回的 **usage** 字段。

**为什么要有这个概念**：只有能计数，才能做预算、限流、截断与成本核算；本 demo 用启发式近似，生产建议用 tokenizer 或 API usage。

### 3. 成本估算

**是什么**：计费通常按「输入 token 单价 + 输出 token 单价」计算。长 system、长历史、长 RAG 注入都会增加输入 token，从而增加单次调用成本。

**为什么要有这个概念**：在加能力（如更长记忆、更多 RAG）时，需要权衡效果与成本；建立「长度 ≈ token ≈ 成本」的意识有助于做架构与产品决策。

---

## 本 Demo 做了什么

- 对示例的 system + user 消息做**字符数**统计，并用简单启发式（如字符数/2）估算**约 token 数**。
- 不实际调用 LLM，仅演示「长度与 token」的关系，并说明生产应使用 tokenizer 或 API usage。

运行：`python -m demos.17_tokens.run`。
