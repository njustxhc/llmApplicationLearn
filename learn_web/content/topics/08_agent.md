# 08 Agent / 智能体

本讲介绍 **Agent**：让大模型不仅「回答一句话」，而是能**自主决定**先查什么、再查什么、何时调用工具、何时给出最终答案，形成多步推理与执行的闭环。本 Demo 提供多种形态：基础循环、**ReAct**（Thought–Action–Observation）、多 MCP Server、以及**规划拆解**（先规划再按步执行），均基于 **core.llm** 与模拟工具环境。

---

## 核心概念

### 1. Agent（智能体）

**是什么**：Agent 指能根据目标与环境反馈自主做决策、执行多步动作的系统。在大模型语境下，通常指：模型根据当前状态和可用工具，决定「下一步做什么」（如调用某工具、或直接回答），执行后把结果作为新信息喂回模型，由模型再决定下一步，直到得出最终答案或结束条件。

**为什么要有这个概念**：单次「一问一答」无法完成「先查天气再查航班再总结」这类任务。Agent 把「规划、调用工具、观察结果、再规划」的循环交给模型驱动，使应用能够完成需要多步推理与外部交互的复杂任务。

### 2. ReAct（Reasoning + Acting）

**是什么**：ReAct 是一种让模型「边想边做」的格式：模型输出 **Thought**（当前思考/计划）→ **Action**（要调用的工具及参数）→ 环境执行后返回 **Observation**（结果）→ 模型再根据 Observation 输出下一个 Thought，如此循环，直到输出最终 **Answer**。

**为什么要有这个概念**：若只让模型一次性列出所有要调用的工具，容易出错且难以根据中间结果调整。ReAct 把推理与行动交错，让模型每步都能根据「上一步的结果」决定下一步，更贴近人类解决问题的方式，也便于调试与可解释性。

### 3. 工具与多 MCP Server

**是什么**：Agent 的「行动」往往通过调用外部工具完成（如查天气、查库、计算）。这些工具可以来自多个 **MCP Server**，即多套工具源；Agent 在每步从「所有已注册工具」中选择一个并传入参数，由运行时执行并返回结果。

**为什么要有这个概念**：现实任务常需组合多种能力（搜索、计算、查库、发邮件等），单一工具集不够用。支持多 MCP Server 能让 Agent 在一个流程里按需使用不同来源的工具，扩展能力边界。

### 4. 规划与执行（Plan-and-Execute）

**是什么**：先让模型根据用户目标生成一个**分步计划**（如步骤 1、2、3…），再按步骤依次执行：每步可能调用工具或生成子回答，把结果作为下一步的输入，直到计划执行完毕并汇总最终答案。

**为什么要有这个概念**：有些任务步骤多、依赖关系清晰，先整体规划再执行，能减少「想到哪做到哪」的混乱，也便于人类审查和干预。规划与执行把「决策」和「执行」在时间上分开，适合流程明确的复杂任务。

### 5. 观察（Observation）与反馈循环

**是什么**：每次工具调用或环境动作后，将执行结果（成功输出或错误信息）作为 **Observation** 反馈给模型。模型把 Observation 纳入上下文，再决定下一轮 Thought 与 Action，形成「决策 → 执行 → 观察 → 再决策」的闭环。

**为什么要有这个概念**：没有观察反馈，模型无法根据「工具实际返回了什么」调整后续动作，容易重复错误或偏离目标。观察是 Agent 与环境交互的桥梁，是实现「根据结果动态调整」的关键。

## 本 Demo 做了什么

### 基础用例

- **run.py**：基础 Agent 循环，根据用户问题与可用工具，进行一次或少量步骤的「决策 → 调用工具 → 观察结果 → 再决策或回答」。帮助理解 Agent 的基本循环与工具调用在 Agent 中的位置。
- 运行：点击「运行基础 Demo」或执行 `python -m demos.08_agent.run`。

### 进阶用例

- **run_react.py**：ReAct（Thought–Action–Observation）模式，显式让模型在每步输出「思考 → 行动（如调用某工具）→ 观察（工具结果）」，多步迭代直到给出最终答案。对比基础用例，进阶更强调「推理链」与「多步工具使用」的可观测性。
- 运行：点击「运行进阶 Demo」或执行 `python -m demos.08_agent.run_react`。其他脚本（run_multi_mcp、run_plan）详见 [demos/08_agent/README.md](../../demos/08_agent/README.md)。
