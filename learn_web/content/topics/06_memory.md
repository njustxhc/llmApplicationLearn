# 06 多轮对话与记忆

本讲介绍如何为对话应用维护**上下文**与**记忆**：既保留多轮对话历史，又能在历史过长时做摘要压缩，并可选地引入「长期记忆」（如用 RAG 存储与检索关键事实）。本 Demo 在 **core.llm** 基础上实现：会话持久化到 `memory.md`、超长时用 LLM 做摘要替换旧消息、以及基于 **core.rag_doc** 的长期记忆检索与注入。

---

## 核心概念

### 1. 对话历史（多轮上下文）

**是什么**：多轮对话时，把历史中的 user 与 assistant 消息按顺序保留在 `messages` 中，每次请求都带上这些历史，使模型能理解「刚才说了什么」「用户指的是哪一句」，从而支持追问、指代、改口等。

**为什么要有这个概念**：若只发当前这一条 user，模型看不到上文，无法做连续对话。对话历史是「记忆」的基线形态，也是后续做摘要、截断、长期记忆的前提。

### 2. 上下文窗口与截断

**是什么**：模型有最大 token 数限制（上下文窗口）。当历史过长超过限制时，必须做截断或压缩：例如只保留最近 N 轮、或丢弃最早的消息、或对旧消息做摘要后再保留。

**为什么要有这个概念**：长对话会撑爆上下文，导致请求失败或性能下降。主动管理「哪些消息保留、以什么形式保留」，才能在长会话中兼顾效果与成本。

### 3. 摘要压缩（Summarization）

**是什么**：当历史消息过多时，用一次 LLM 调用对「较早的若干轮」做摘要，得到一段缩短的总结文本，再用这段总结替代原来的多轮消息。这样既保留关键信息，又节省 token。

**为什么要有这个概念**：简单截断会直接丢失前文信息；全保留又容易超长。用摘要把「旧对话」压缩成一段话，既控制长度又保留大意，是长会话中常用的折中方案。

### 4. 会话持久化

**是什么**：把当前会话的 messages（或摘要后的版本）写入本地文件或数据库（本 Demo 用 `memory.md`），下次启动或下次进入同一会话时再读入，使对话可以跨进程、跨时间延续。

**为什么要有这个概念**：不持久化的话，关掉应用或刷新页面后历史就没了。持久化后可以支持「继续上次对话」、多设备同步、或作为长期记忆的数据源。

### 5. 长期记忆（Long-term Memory）与 RAG

**是什么**：除了「当前会话历史」，还可以把对话中的关键事实、用户偏好等抽取出来，存入独立的知识库（如 `memories.md` 或向量库），之后在新对话中通过检索（如 RAG）把相关记忆注入上下文，实现「记得你之前说过……」的效果。

**为什么要有这个概念**：单次会话的历史再长也有上限，且会随摘要/截断丢失细节。把重要信息沉淀到长期记忆并支持检索，能让模型在多次会话、多轮对话中保持「跨会话」的连续性，更贴近真实使用体验。

---

## 本 Demo 做了什么

- **会话持久化**：将当前会话保存到 `demos/06_memory/memory.md`（Markdown 格式），下次运行可加载继续。
- **超长摘要**：当消息条数超过设定阈值时，用 **core.llm** 对较早消息做摘要，用摘要结果替换旧消息，再保留最近若干轮原文。
- **长期记忆**：从对话中抽取关键事实写入 `long_term/memories.md`，回答前用 **core.rag_doc** 按问题检索相关记忆片段并注入 prompt，实现「长期记忆」增强。
- **交互方式**：在终端中运行，从终端读取用户输入并循环对话（学习站内不直接执行，会提示在终端运行 `python -m demos.06_memory.run`）。

运行：在项目根目录执行 `python -m demos.06_memory.run`，按提示输入内容进行多轮对话。
