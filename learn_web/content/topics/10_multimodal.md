# 10 多模态（图/音）

本讲介绍 **多模态**：大模型不仅能处理文本，还能处理图像、语音等模态；输入或输出可以是「文本 + 图片」「语音转文字」「文字转语音」等组合。本 Demo 在 **core.llm** 基础上说明多模态消息的常见结构（如 `content` 为 list，含 `type: text` 与 `type: image_url`），并演示当前可用的文本调用；实际接入图像/语音需按所用 API（如火山方舟多模态文档）补充请求格式与端点。

---

## 核心概念

### 1. 多模态（Multimodal）

**是什么**：多模态指模型能同时处理多种类型的数据输入或输出，例如文本、图像、音频。在对话中，一条消息的 `content` 可以是混合的：既有文本块，也有图片 URL 或 base64 图片，模型根据文本与图像共同理解并生成回复。

**为什么要有这个概念**：很多场景需要「看图说话」或「听音转文」：客服传截图、教育题配图、语音助手等。多模态把文本与非文本统一到同一套接口与上下文里，扩展了模型的应用边界。

### 2. 消息结构中的多模态 content

**是什么**：在多模态 API 中，单条消息的 `content` 往往是一个列表，每项标明类型与内容，例如：`{ "type": "text", "text": "请描述这张图" }`、`{ "type": "image_url", "image_url": { "url": "..." } }`。模型会同时看到文本与图片，并可能输出文本或（若 API 支持）图像/语音。

**为什么要有这个概念**：纯文本的 `content` 无法表达图片或语音；用结构化 content 列表能在一轮消息里混合多种模态，便于 API 设计与解析，也是多模态调用的标准形态。

### 3. Vision（视觉）与图像理解

**是什么**：Vision 指模型能「看」图像并理解内容、回答问题、或根据图文生成描述/代码。通常通过把图像编码或 URL 放入消息的 image 类 content 实现，模型在内部对齐图文表示后生成回复。

**为什么要有这个概念**：文档、截图、图表、实物照片等大量信息以图像形式存在；仅靠文本无法利用这些信息。支持 Vision 后，RAG、客服、教育等场景可以直接「问图」「以图搜图」或「图+文」联合检索与生成。

### 4. 语音：ASR 与 TTS

**是什么**：**ASR（语音识别）**：把语音转成文字，作为 user 消息输入模型。**TTS（文本转语音）**：把模型输出的文字转成语音播放。多模态链路中，语音往往先转为文本再进 LLM，或 LLM 输出文本后再转为语音。

**为什么要有这个概念**：语音是许多交互场景的主入口（智能音箱、车载、客服）。把语音纳入「多模态」范畴，便于在设计时统一考虑输入输出形态；实际实现时通常依赖专门的 ASR/TTS API 或模型。

### 5. 多模态与 RAG / Agent

**是什么**：在多模态基础上，RAG 可以检索「图文混合」文档或图片描述；Agent 可以接收用户上传的图片、根据图像内容决定调用哪些工具或生成什么回答。多模态扩展了 RAG 与 Agent 的输入形态与能力边界。

**为什么要有这个概念**：真实业务中常同时存在文本与图像数据；多模态 RAG 与多模态 Agent 能更好地利用这些数据，提供更完整的应用体验。

---

## 本 Demo 做了什么

- 说明多模态消息的常见结构（如 `content` 为 list，含 `type: text` 与 `type: image_url`），便于后续接入真实多模态 API。
- 当前仅演示基于 **core.llm** 的文本调用；实际接入图像/语音需参考火山方舟等多模态文档，在请求中按规范构造多模态 content 并调用对应端点。

运行：在项目根目录执行 `python -m demos.10_multimodal.run`。
