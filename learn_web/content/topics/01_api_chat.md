# 01 基础 API 与对话

本讲通过一次最简单的「发一条消息、收一条回复」来熟悉大模型 API 的调用方式，并理解多轮对话所依赖的 **messages** 结构。本 Demo 使用项目统一的 **core.llm**（火山方舟 Ark API）完成一次单轮问答。

---

## 核心概念

### 1. 大模型 API（Chat Completions）

**是什么**：大模型以「对话补全」形式对外提供服务：你发送一段或多段对话内容（通常包含角色与文本），接口返回模型生成的下一段回复。业界常见形态是「Chat Completions」或「对话补全」API：请求体里是 `messages` 列表，响应里是 `choices[0].message` 的文本。

**为什么要有这个概念**：应用层所有对话、工具调用、多轮记忆都建立在「发 messages、收 completion」这一层之上。先掌握这一层，才能理解后面的 system/user 分工、多轮历史、流式输出等。

### 2. messages 与角色（system / user / assistant）

**是什么**：`messages` 是一组按顺序排列的消息，每条有 `role` 和 `content`。常见角色包括：**system**（系统设定，如身份、规则）、**user**（用户输入）、**assistant**（模型历史回复）。接口根据整段 messages 预测「下一条」assistant 内容。

**为什么要有这个概念**：角色区分了「谁在说话」以及「这段内容起什么作用」。system 用来固定模型行为，user/assistant 构成对话历史；多轮对话、RAG 注入、工具结果回填，都是往 messages 里追加或插入内容，因此这是整个对话架构的基础。

### 3. 流式（stream）与非流式

**是什么**：**非流式**：请求发出去后，等服务端生成完整回复再一次性返回。**流式**：服务端边生成边返回（如 SSE），客户端可以逐 token 展示，降低首字延迟、提升体验。

**为什么要有这个概念**：长回答时非流式等待时间长，用户容易以为卡死；流式能即时看到输出，更适合聊天产品。同时流式在实现上要处理分块、拼接和结束标记，是工程上常见需求。

### 4. 多轮对话（对话历史）

**是什么**：多轮对话指把「上一轮的用户问题 + 模型回答」也放进下一次请求的 `messages` 里，让模型在生成新回复时能看到之前说了什么，从而具备上下文连续、指代消解、追问回答等能力。

**为什么要有这个概念**：单轮只能一问一答；真实产品里用户会追问、改口、引用上文。把历史纳入 messages 是实现「连续对话」的标准做法，也是后续做记忆、摘要、上下文窗口管理的前提。

## 本 Demo 做了什么

### 基础用例

- 使用 **core.llm** 的 `chat(messages)` 发一条 user 消息，收一条 assistant 回复。
- 不设 system、不保留历史，仅演示最基础的「一问一答」调用链。
- 运行：点击「运行基础 Demo」或执行 `python -m demos.01_api_chat.run`。

### 进阶用例

- 在同一会话中连续进行 3 轮「用户提问 → 模型回复」，每轮将上一轮的 user 与 assistant 追加到 `messages` 再请求，演示多轮对话与上下文。
- 运行：点击「运行进阶 Demo」或执行 `python -m demos.01_api_chat.run_advanced`。
