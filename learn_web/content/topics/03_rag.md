# 03 检索增强（RAG）

本讲介绍 **RAG（Retrieval-Augmented Generation）**：在回答用户问题前，先从知识库中检索相关片段，再把「检索结果 + 用户问题」一起交给大模型，让回答有据可依、减少幻觉。本 Demo 使用 **core.rag_doc** 做文档检索，再配合 **core.llm** 生成答案。

---

## 核心概念

### 1. RAG（检索增强生成）

**是什么**：RAG 是一种「先检索、再生成」的流程：用户提问 → 用检索系统（如关键词、向量）从文档/知识库中找出相关片段 → 把这些片段和问题一起作为上下文喂给大模型 → 模型在「有依据」的上下文中生成答案。

**为什么要有这个概念**：大模型的知识有训练截止日期、且容易「一本正经地胡说」。把外部知识通过检索注入到 prompt 里，既能利用模型的理解与生成能力，又能把答案锚定在真实文档上，适合企业知识库、说明书、法规等场景。

### 2. 检索（Retrieval）

**是什么**：根据用户问题，从大量文档中找出「最相关」的若干片段（如段落、句子）。常见做法包括：关键词匹配（如 TF-IDF、BM25）、向量相似度（Embedding + 向量库）、或二者结合。

**为什么要有这个概念**：不检索的话，要么模型只靠自身知识（易过时、易幻觉），要么把整库塞进 prompt（超长、超贵、超慢）。检索把「大海捞针」交给专门的检索模块，只把少量相关片段交给 LLM，成本与效果更可控。

### 3. 文档切块（Chunking）

**是什么**：长文档在入库前会被切成较小单元（块），如按段落、按固定长度、按语义段。检索时以「块」为单位返回，避免整篇文档塞进一条消息。

**为什么要有这个概念**：模型有上下文长度上限；且检索粒度太粗（整篇）往往噪声大、相关性差。按块存储与检索，能在控制 token 用量的前提下，尽量保留与问题最相关的那几段。

### 4. 上下文注入（Context Injection）

**是什么**：把检索到的文本作为「上下文」拼进发给模型的 prompt 里，通常放在 system 或 user 消息中，例如：「根据以下内容回答：…… [检索结果] …… 用户问题：……」。

**为什么要有这个概念**：模型不会主动「去查库」，只有把检索结果显式写进 messages，模型才能基于这些内容作答。注入方式（放 system 还是 user、格式、长度）会直接影响答案质量和幻觉程度。

## 本 Demo 做了什么

### 基础用例

- 针对一个固定问题（如「这个项目有哪些功能？」），调用 **core.rag_doc** 的 `get_rag_context(query, top_k=3)` 做一次文档检索，将得到的文本片段作为 system 或上下文注入到 messages，再调用 **core.llm** 生成有依据的答案并打印。
- 演示「先检索、再生成」的 RAG 基本流程与上下文注入方式。
- 运行：点击「运行基础 Demo」或执行 `python -m demos.03_rag.run`。

### 进阶用例

- 在单次检索基础上扩展：进行「多路检索」或「带引用标注」——在答案中标注每段依据来自哪条检索片段；或用更大 top_k 检索并在 system 中要求模型回答时注明来源。
- 运行：点击「运行进阶 Demo」或执行 `python -m demos.03_rag.run_advanced`。
